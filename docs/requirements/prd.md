# Product Requirements Document (PRD)

## 1. Обзор продукта

### 1.1 Описание
Мобильное приложение для клиентов управляющей компании, позволяющее просматривать инвестиционные продукты, отслеживать портфель активов и общаться с персональным менеджером через in-app чат.

### 1.2 Цель продукта
Предоставить состоятельным инвесторам удобный инструмент для мониторинга и управления своими активами, переданными в доверительное управление, обеспечивая прозрачность и постоянную связь с персональным менеджером.

### 1.3 Статус
**Draft** | Дата создания: 2026-04-02

### 1.4 Платформы
- iOS (React Native)
- Android (React Native)

---

## 2. Целевая аудитория

### 2.1 Основной сегмент
**Состоятельные инвесторы** - физические лица с высоким уровнем дохода и накоплений, которые:
- Передают активы или часть активов в доверительное управление
- Имеют минимальный порог инвестиций (определяется управляющей компанией)
- Ценят персонализированный подход и постоянную связь с менеджером
- Ожидают высокого уровня сервиса и прозрачности

### 2.2 Характеристики пользователей
- **Возраст:** 35-65 лет
- **Образование:** Высшее, часто с экономическим/финансовым бэкграундом
- **Технологическая грамотность:** Средняя-высокая
- **Поведение:** Предпочитают мобильные приложения для повседневных задач
- **Потребности:** Прозрачность инвестиций, быстрый доступ к информации, персональное обслуживание

## 3. Персоны

### Персона 1: Александр, 48 лет
**Бизнесмен-инвестор**

**Профиль:**
- Владелец среднего бизнеса
- Инвестировал 50+ млн руб. в доверительное управление
- Активно следит за рынками, но не имеет времени на детальный анализ
- Имеет персонального менеджера в управляющей компании

**Цели:**
- Быстро проверить текущее состояние портфеля
- Оценить доходность по стратегиям
- Обсудить с менеджером возможность новых инвестиций

**Боли:**
- Неудобно звонить менеджеру по телефону для каждого вопроса
- Хочет видеть актуальную информацию в любое время
- Трудно отслеживать несколько стратегий одновременно

**Сценарий использования:**
Открывает приложение утром, просматривает портфель, видит уведомление о новой стратегии, пишет менеджеру в чат для уточнения деталей.

### Персона 2: Елена, 52 года
**Инвестор-рантье**

**Профиль:**
- На пенсии, живёт на доход от инвестиций
- Передала активы в доверительное управление 3 года назад
- Менее активный инвестор, предпочитает консервативные стратегии
- Ценит стабильность и прозрачность

**Цели:**
- Увидеть, какой доход приносят инвестиции
- Проверить безопасность активов
- Получить консультацию менеджера при необходимости

**Боли:**
- Сложно понимать финансовые термины
- Беспокоится о сохранности капитала
- Хочет простого и понятного интерфейса

**Сценарий использования:**
Раз в неделю открывает приложение, просматривает портфель, видит рост капитала, чувствует спокойствие. При вопросах пишет менеджеру в чат.

## 6. Архитектура

### 6.1 Трёхслойная архитектура

Приложение строится на трёхслойной архитектуре для обеспечения разделения ответственности и тестируемости:

#### Слой 1: View (Presentation Layer)
**Ответственность:**
- Отображение UI экранов
- Обработка пользовательского ввода
- Навигация между экранами
- Анимации и визуальные эффекты

**Технологии:**
- React Native компоненты
- Tailwind CSS для стилизации
- shadcn/ui компоненты

**Структура:**
```
src/
  screens/          # Экраны приложения
    Auth/
    Products/
    Portfolio/
    Chat/
    Profile/
  components/       # Переиспользуемые компоненты
    common/
    products/
    portfolio/
    chat/
  navigation/       # Конфигурация навигации
```

#### Слой 2: Domain (Business Logic Layer)
**Ответственность:**
- Определение бизнес-сущностей (модели данных)
- Бизнес-логика и правила валидации
- Use Cases (сценарии использования)
- Интерфейсы репозиториев

**Структура:**
```
src/
  domain/
    entities/       # Модели данных
      User.ts
      Product.ts
      Portfolio.ts
      Asset.ts
      Message.ts
    usecases/       # Сценарии использования
      GetProductsUseCase.ts
      GetPortfolioUseCase.ts
      SendMessageUseCase.ts
    repositories/   # Интерфейсы репозиториев
      IProductRepository.ts
      IPortfolioRepository.ts
      IChatRepository.ts
```

#### Слой 3: Adapters (Data Layer)
**Ответственность:**
- Интеграция с бэкендом (API клиент)
- Кэширование данных
- Маппинг данных между API и Domain
- Реализация репозиториев

**Структура:**
```
src/
  adapters/
    api/           # API клиент
      ApiClient.ts
      AuthApi.ts
      ProductsApi.ts
      PortfolioApi.ts
      ChatApi.ts
    storage/       # Локальное хранилище
      SecureStorage.ts
      CacheStorage.ts
    repositories/  # Реализация репозиториев
      ProductRepository.ts
      PortfolioRepository.ts
      ChatRepository.ts
```

### 6.2 Поток данных

```
View → UseCase → Repository (Interface) → Repository (Implementation) → API/Storage
                                    ↓
                               Domain Entities
```

### 6.3 Dependency Injection

Использование Inversion of Control (IoC) для внедрения зависимостей:
- Repository интерфейсы инжектируются в Use Cases
- Use Cases инжектируются в View Models/Components
- Упрощает тестирование и замену реализаций

## 7. Технический стек

### 7.1 Frontend
- **Фреймворк:** React Native 0.73+
- **Язык:** TypeScript 5.0+
- **Навигация:** React Navigation 6.x
- **State Management:** React Query (TanStack Query) + Zustand
- **Стилизация:** Tailwind CSS (twrnc или NativeWind)
- **UI Kit:** shadcn/ui (адаптированный для React Native)
- **Графики:** Victory Native или React Native SVG Charts
- **Иконки:** Lucide React Native

### 7.2 Backend
- **Runtime:** Node.js 18+ LTS
- **Framework:** Express.js или Fastify
- **База данных:** PostgreSQL
- **Кэш:** Redis
- **Message Queue:** RabbitMQ (для чата)
- **WebSockets:** Socket.io (для real-time чата)

### 7.3 Инфраструктура
- **CI/CD:** GitHub Actions
- **Мониторинг:** Sentry (ошибки), Datadog/New Relic (производительность)
- **Аналитика:** Amplitude или Mixpanel
- **Push-уведомления:** Firebase Cloud Messaging + APNs

### 7.4 Безопасность
- **Аутентификация:** JWT + Refresh Tokens
- **Шифрование:** crypto-js (client), bcrypt (server)
- **Сертификаты:** Let's Encrypt + Cloudflare

### 7.5 Тестирование
- **Unit тесты:** Jest + React Testing Library
- **E2E тесты:** Detox или Appium
- **API тесты:** Postman / Newman

## 10. За рамками проекта (Out of Scope)

### 10.1 Функциональность, исключённая из MVP
- **Торговые функции:** Покупка/продажа активов напрямую через приложение
- **Вывод средств:** Инициация вывода средств через приложение
- **Самостоятельная регистрация:** Все клиенты приглашаются менеджерами
- **Социальные функции:** Форумы, комментарии, рейтинги стратегий
- **Robo-advisor:** Автоматические инвестиционные рекомендации на основе ИИ
- **Образовательный контент:** Статьи, видео, вебинары (может быть добавлено позже)

### 10.2 Технические ограничения MVP
- Нет офлайн-режима для чата (только просмотр кэшированной истории)
- Нет синхронизации данных между устройствами в реальном времени
- Нет поддержки iPad-специфичного интерфейса (только масштабирование)
- Нет виджетов для домашнего экрана

## 11. Риски и ограничения

### 11.1 Технические риски
| Риск | Вероятность | Влияние | Митигация |
|------|------------|---------|-----------|
| Проблемы с интеграцией бэкенда | Средняя | Высокое | Раннее тестирование API, мок-данные для разработки |
| Проблемы производительности на старых устройствах | Средняя | Среднее | Оптимизация графиков, ленивая загрузка |
| Сложности с push-уведомлениями на Android | Низкая | Среднее | Использование проверенных библиотек, тестирование на разных устройствах |

### 11.2 Бизнес-риски
| Риск | Вероятность | Влияние | Митигация |
|------|------------|---------|-----------|
| Низкая адаптация пользователями | Средняя | Высокое | Обучение клиентов, демонстрация ценности приложения |
| Недостаток ресурсов бэкенд-команды | Высокая | Высокое | Чёткое планирование API, приоритизация функций |
| Изменения требований регулятора | Низкая | Высокое | Консультации с юристами, гибкая архитектура |

### 11.3 Ограничения
- **Бюджет:** Определяется на этапе планирования спринтов
- **Сроки:** MVP - 3-4 месяца (зависит от ресурсов бэкенд-команды)
- **Ресурсы:** 2 frontend-разработчика (React Native), 1 backend-разработчик (part-time), 1 QA (part-time)
- **Зависимости:** Готовность бэкенда и API, наличие тестовых данных

## 12. Дорожная карта (Roadmap)

### Фаза 1: MVP (Месяцы 1-3)
**Цель:** Запуск минимально жизнеспособного продукта

**Месяц 1:**
- Настройка проекта и архитектуры
- Реализация аутентификации и базовой навигации
- Интеграция с API бэкенда (мок-данные)

**Месяц 2:**
- Разработка экрана витрины продуктов
- Разработка экрана портфеля активов
- Интеграция с реальным бэкендом

**Месяц 3:**
- Разработка чата с менеджером
- Реализация push-уведомлений
- Тестирование и исправление дефектов
- Подготовка к публикации в сторах

- Оптимизация производительности

**Цель:** Добавление продвинутых функций

- Английская локализация

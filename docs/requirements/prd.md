# Product Requirements Document (PRD)

## 1. Обзор продукта

### 1.1 Название продукта
Мобильное приложение для клиентов управляющей компании (внутреннее название: Asset Management Client App)

### 1.2 Описание
Мобильное приложение для платформ iOS и Android, разработанное на базе React Native. Приложение предназначено для клиентов управляющей компании и предоставляет удобный доступ к информации о продуктах доверительного управления, текущему портфелю активов и коммуникации с персональным менеджером.

### 1.3 Статус
Draft

### 1.4 Дата создания
2026-04-08

### 1.5 Версия документа
1.0

---

## 2. Цели и задачи

### 2.1 Бизнес-цели
- **Повышение клиентоориентированности**: Обеспечить клиентам удобный мобильный доступ к информации об их активах 24/7
- **Улучшение коммуникации**: Создать прямой канал связи между клиентами и персональными менеджерами
- **Увеличение продаж продуктов**: Продемонстрировать витрину продуктов для стимулирования дополнительных инвестиций
- **Конкурентное преимущество**: Современное мобильное приложение как фактор привлечения новых клиентов

### 2.2 Продуктовые цели
- Создать интуитивно понятный интерфейс для просмотра продуктов компании
- Обеспечить прозрачность портфеля активов клиента в реальном времени
- Наладить эффективную коммуникацию клиент-менеджер через in-app чат

### 2.3 Ключевые метрики успеха (KPI)
- Количество активных пользователей (MAU - Monthly Active Users)
- Время, проведенное в приложении на пользователя
- Количество сообщений в чатах в месяц
- Конверсия просмотра продуктов в заявки на инвестирование
- Рейтинг приложения в App Store и Google Play

## 4. Функциональные требования

### 4.1 Must Have (P0) - Критически важно для MVP

#### FR-001: Аутентификация и авторизация
**Описание**: Безопасный вход в приложение для зарегистрированных клиентов
**Требования**:
- Вход по номеру телефона + SMS-код
- Вход по email + пароль (альтернативный вариант)
- Поддержка биометрической аутентификации (Face ID / Touch ID) для быстрого входа
- Автоматический выход после 30 минут неактивности
- Возможность выхода из всех устройств

**Критерии приемки**:
- Пользователь может войти в приложение используя телефон или email
- Биометрический вход работает корректно на поддерживаемых устройствах
- Сессия автоматически завершается после 30 минут бездействия

#### FR-002: Витрина продуктов
**Описание**: Экран со списком доступных продуктов доверительного управления
**Требования**:
- Отображение списка стратегий доверительного управления
- Карточка продукта с названием, описанием, доходностью
- Детальный просмотр каждого продукта
- Информация о стратегии индивидуального доверительного управления
- Фильтрация продуктов по категориям/уровню риска
- Отображение исторической доходности (графики)

**Критерии приемки**:
- Пользователь видит все доступные продукты компании
- Можно открыть детальную информацию о продукте
- Графики доходности отображаются корректно

#### FR-003: Портфель активов клиента
**Описание**: Экран с текущим портфелем пользователя под управлением
**Требования**:
- Общая стоимость портфеля
- Структура портфеля (круговая диаграмма по классам активов)
- Список активов с текущей стоимостью
- Доходность портфеля (общая, за период)
- Детализация по каждому активу/стратегии
- История изменений портфеля
- Возможность выбора периода анализа (1 месяц, 3 месяца, 6 месяцев, 1 год, все время)

**Критерии приемки**:
- Пользователь видит актуальную стоимость портфеля
- Можно просмотреть детализацию по каждому активу
- Графики доходности отображаются корректно

#### FR-004: In-app чат с персональным менеджером
**Описание**: Экран с чатом для коммуникации с менеджером
**Требования**:
- Отображение имени и фото менеджера
- Отправка текстовых сообщений
- Получение уведомлений о новых сообщениях
- История переписки
- Статус доставки сообщений (отправлено, доставлено, прочитано)
- Возможность отправки вложений (документы, изображения)

**Критерии приемки**:
- Пользователь может отправить сообщение менеджеру
- Уведомления о новых сообщениях приходят вовремя
- История чата сохраняется

### 4.2 Should Have (P1) - Важно для первой версии

#### FR-005: Push-уведомления
**Описание**: Уведомления о важных событиях
**Требования**:
- Уведомления о новых сообщениях в чате
- Уведомления об изменениях в портфеле
- Уведомления о новых продуктах
- Настройка типов уведомлений

**Критерии приемки**:
- Push-уведомления приходят корректно
- Пользователь может настроить типы уведомлений

#### FR-006: Профиль пользователя
**Описание**: Экран с информацией о клиенте и настройках
**Требования**:
- Отображение имени, контактов
- Настройки уведомлений
- Настройки безопасности (смена пароля, биометрия)
- Политика конфиденциальности и условия использования
- Кнопка выхода из аккаунта

**Критерии приемки**:
- Пользователь может изменить настройки
- Выход из аккаунта работает корректно

#### FR-007: Заявка на продукт
**Описание**: Возможность подать заявку на инвестирование в продукт
**Требования**:
- Кнопка "Подать заявку" на странице продукта
- Форма заявки с указанием суммы
- Отправка заявки менеджеру

**Критерии приемки**:
- Заявка отправляется менеджеру
- Пользователь получает подтверждение

### 4.3 Could Have (P2) - Желательно в будущих версиях

#### FR-008: Аналитика и отчеты
**Описание**: Расширенная аналитика портфеля
**Требования**:
- Сравнение доходности с бенчмарками
- Экспорт отчетов в PDF
- Детальная аналитика по рискам

#### FR-009: Новости и аналитика рынка
**Описание**: Информационный раздел
**Требования**:
- Лента новостей от компании
- Аналитические обзоры рынка
- Рекомендации менеджера

#### FR-010: Мультиязычность
**Описание**: Поддержка нескольких языков
**Требования**:
- Русский язык (основной)
- Английский язык (опционально)

### 4.4 Won't Have (P3) - Не планируется в текущей версии

#### FR-011: Торговые операции
Самостоятельное совершение сделок покупки/продажи активов (только через менеджера)

#### FR-012: Интеграция с внешними сервисами
Подключение внешних банковских счетов, других брокеров

#### FR-013: Социальные функции
Рейтинги менеджеров, отзывы клиентов, сообщества инвесторов

## 5. Нефункциональные требования

### 5.1 Производительность (Performance)
- **NFR-001**: Время запуска приложения не более 3 секунд
- **NFR-002**: Загрузка экранов не более 2 секунд
- **NFR-003**: Поддержка offline-режима для ранее загруженных данных
- **NFR-004**: Оптимизация потребления батареи

### 5.2 Безопасность (Security)
- **NFR-005**: Шифрование данных при передаче (HTTPS/TLS 1.3)
- **NFR-006**: Шифрование локальных данных устройства
- **NFR-007**: Защита от MITM-атак (Certificate Pinning)
- **NFR-008**: Соответствие требованиям регулятора (ЦБ РФ для финансовых приложений)
- **NFR-009**: Логирование действий безопасности
- **NFR-010**: Автоматический logout при Root/Jailbreak устройства

### 5.3 Надежность (Reliability)
- **NFR-011**: Доступность приложения 99.5%
- **NFR-012**: Graceful degradation при недоступности бэкенда
- **NFR-013**: Автоматическое восстановление после сбоев

### 5.4 Масштабируемость (Scalability)
- **NFR-014**: Поддержка до 10,000 одновременных пользователей
- **NFR-015**: Архитектура, позволяющая горизонтальное масштабирование

### 5.5 Совместимость (Compatibility)
- **NFR-016**: iOS 14.0 и выше
- **NFR-017**: Android 8.0 (API 26) и выше
- **NFR-018**: Адаптация под различные размеры экранов (телефоны и планшеты)

### 5.6 Удобство использования (Usability)
- **NFR-019**: Интуитивно понятный интерфейс без необходимости обучения
- **NFR-020**: Соответствие гайдлайнам Human Interface Guidelines (iOS) и Material Design (Android)
- **NFR-021**: Поддержка динамического размера шрифта (Accessibility)

### 5.7 Поддержка и сопровождение
- **NFR-022**: Возможность обновления без пересмотра приложения (remote config)
- **NFR-023**: Crash reporting и аналитика использования
- **NFR-024**: Feature flags для поэтапного внедрения функций

## 6. Технические требования

### 6.1 Архитектура
Приложение должно иметь трехслойную архитектуру:

#### Слой View (Presentation Layer)
- Компоненты экранов и UI-элементы
- Навигация между экранами
- Обработка пользовательского ввода
- Отображение данных из Domain слоя

#### Слой Domain (Business Logic Layer)
- Сущности (Entities): User, Portfolio, Asset, Product, Message
- Use Cases / Interactors: бизнес-логика приложения
- Repository Interfaces: контракты для работы с данными
- Валидация и бизнес-правила

#### Слой Adapters (Data Layer)
- Repository Implementations: реализация репозиториев
- API Clients: интеграция с бэкендом
- Local Storage: кэширование данных на устройстве
- Модели данных (DTO)

### 6.2 Технологический стек

| Компонент | Технология |
|-----------|------------|
| Mobile Framework | React Native |
| Язык программирования | TypeScript |
| UI Components | shadcn/ui |
| Styling | Tailwind CSS (NativeWind) |
| State Management | TBD (Redux Toolkit / Zustand) |
| Navigation | React Navigation |
| Backend | Node.js |
| API Protocol | REST / GraphQL (TBD) |
| Authentication | JWT + OAuth 2.0 |
| Database (Mobile) | AsyncStorage / SQLite |
| Push Notifications | Firebase Cloud Messaging |
| Analytics | Firebase Analytics / Amplitude |
| Crash Reporting | Sentry |

### 6.3 API и интеграции
- **Backend API**: RESTful API или GraphQL для синхронизации данных
- **Authentication Service**: Сервис авторизации управляющей компании
- **Push Notifications**: Firebase Cloud Messaging / Apple Push Notifications
- **Analytics**: Аналитическая платформа для отслеживания метрик

## 7. Ограничения и допущения

### 7.1 Ограничения
- Приложение разрабатывается только для платформ iOS и Android
- MVP должен быть готов в течение 4-6 месяцев
- Бюджет определяется индивидуально
- Нет возможности самостоятельных торговых операций клиентом
- Требуется соответствие требованиям ЦБ РФ к финансовым приложениям

### 7.2 Допущения
- Бэкенд API уже существует или разрабатывается параллельно
- Управляющая компания предоставляет дизайн-систему или бренд-гайды
- Есть доступ к тестовым аккаунтам клиентов для QA
- Персональные менеджеры имеют систему для работы с чатом (web-интерфейс)

### 7.3 Зависимости
- Готовность бэкенд API
- Предоставление данных о продуктах и портфелях в структурированном виде
- Система авторизации существующих клиентов
- Канал связи с персональными менеджерами

### 9.1 MVP Критерии (фаза 1)
- ✅ Приложение опубликовано в App Store и Google Play
- ✅ Аутентификация работает корректно
- ✅ Витрина продуктов отображается с актуальными данными

- NPS (Net Promoter Score) 30+

### 9.3 Технические критерии
- Crash-free rate > 99%
- Среднее время ответа API < 500ms
- Zero critical security vulnerabilities
- Code coverage > 60%

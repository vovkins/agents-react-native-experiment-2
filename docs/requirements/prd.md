# Product Requirements Document (PRD)

## 1. Обзор продукта

### 1.1 Название продукта
Мобильное приложение для клиентов управляющей компании (название TBD)

### 1.2 Описание
Мобильное приложение на базе React Native для платформ iOS и Android, предназначенное для клиентов управляющей компании. Приложение предоставляет состоятельным инвесторам доступ к информации о продуктах доверительного управления, текущему портфелю активов и обеспечивает коммуникацию с персональным менеджером через in-app чат.

### 1.3 Статус
Draft

### 1.4 Дата создания
2026-04-13

### 1.5 Версия документа
1.0

---

## 2. Цели и задачи

### 2.1 Бизнес-цели
- Повышение удовлетворенности клиентов за счет удобного доступа к информации об их активах
- Улучшение коммуникации между клиентами и персональными менеджерами
- Увеличение прозрачности управления активами
- Снижение операционной нагрузки на менеджеров через цифровой канал связи

### 2.2 Продуктовые цели
- Создать интуитивно понятный интерфейс для просмотра инвестиционных продуктов
- Обеспечить实时 доступ к информации о портфеле активов
- Реализовать удобный канал связи с персональным менеджером

### 2.3 Ключевые метрики успеха (KPI)
- MVP готов к тестированию в установленные сроки
- Все основные функции работают корректно
- Код покрыт тестами
- Уровень удовлетворенности пользователей (NPS) после пилотного запуска

## 3. Целевая аудитория

### 3.1 Описание
Состоятельные инвесторы, передающие свои активы или часть активов в доверительное управление управляющей компании.

### 3.2 Характеристики аудитории
- Высокий уровень дохода
- Интерес к инвестиционным продуктам и стратегиям управления активами
- Потребность в персонализированном обслуживании
- Ожидание высокого уровня сервиса и безопасности

### 3.3 Персоны

#### Персона 1: Опытный инвестор
- **Имя:** Александр
- **Возраст:** 45-55 лет
- **Профиль:** Успешный предприниматель с опытом инвестирования
- **Потребности:** Детальная аналитика по портфелю, быстрый доступ к менеджеру, информация о новых стратегиях
- **Боли:** Недостаток прозрачности в управлении активами, сложность коммуникации с менеджером
- **Мотивация:** Контроль над инвестициями, оптимизация доходности

#### Персона 2: Новичок в доверительном управлении
- **Имя:** Елена
- **Возраст:** 35-45 лет
- **Профиль:** Руководитель среднего звена, новичок в доверительном управлении
- **Потребности:** Понятное описание продуктов, образовательные материалы, поддержка менеджера
- **Боли:** Сложность понимания инвестиционных терминов, неуверенность в выборе стратегии
- **Мотивация:** Сохранение и приумножение капитала с профессиональной помощью

## 4. Функциональные требования

### 4.1 Приоритизация MoSCoW

### MUST HAVE (Обязательно)

#### FR-001: Аутентификация и авторизация
- Регистрация через приложение (после заключения договора с компанией)
- Вход в систему (логин/пароль)
- Возможность восстановления пароля
- Автоматический выход после периода неактивности (требует уточнения)

#### FR-002: Витрина продуктов
- Просмотр списка доступных стратегий доверительного управления
- Просмотр детальной информации по каждой стратегии
- Просмотр продукта по индивидуальному доверительному управлению
- Отображение ключевых характеристик продуктов (доходность, риск, минимальная сумма инвестиций)

#### FR-003: Портфель активов
- Отображение текущего портфеля активов под управлением
- Просмотр структуры портфеля (распределение по активам)
- Отображение текущей стоимости портфеля
- История изменений портфеля

#### FR-004: Чат с персональным менеджером
- Отправка текстовых сообщений менеджеру
- Получение сообщений от менеджера
- История переписки
- Уведомления о новых сообщениях

### SHOULD HAVE (Желательно)

#### FR-005: Push-уведомления
- Уведомления о новых сообщениях в чате
- Уведомления об изменениях в портфеле
- Уведомления о новых продуктах

#### FR-006: Биометрическая аутентификация
- Вход с помощью Touch ID / Face ID (iOS)
- Вход с помощью отпечатка пальца / Face Unlock (Android)

#### FR-007: Профиль пользователя
- Просмотр личной информации
- Редактирование контактных данных
- Настройки уведомлений

### COULD HAVE (Возможно)

#### FR-008: Документы и отчеты
- Просмотр документов по портфелю
- Скачивание отчетов в PDF
- Архив документов

#### FR-009: Графики и аналитика
- Визуализация динамики портфеля
- Графики доходности
- Сравнение с бенчмарками

#### FR-010: Образовательный контент
- Статьи об инвестициях
- FAQ раздел
- Глоссарий терминов

### WON'T HAVE (Не входит в MVP)

#### FR-011: Торговые операции
- Покупка/продажа активов через приложение
- Самостоятельное управление портфелем

#### FR-012: Интеграция с внешними сервисами
- Банковские переводы
- Внешние платежные системы

## 5. Нефункциональные требования

### 5.1 Производительность
- Время загрузки главного экрана: не более 2 секунд
- Время отклика на действия пользователя: не более 500 мс
- Поддержка работы в офлайн-режиме для кэшированных данных

### 5.2 Безопасность
- Шифрование данных при передаче (HTTPS/TLS)
- Шифрование чувствительных данных на устройстве
- Безопасное хранение учетных данных
- Защита от основных типов атак (XSS, SQL injection на уровне API)
- Соответствие требованиям регуляторов (требует уточнения)

### 5.3 Доступность
- Поддержка экранов чтения с экрана (VoiceOver, TalkBack)
- Минимальный размер шрифта для удобного чтения
- Контрастность интерфейса согласно WCAG 2.1

### 5.4 Надежность
- Доступность приложения: 99.5%
- Обработка ошибок сети с понятными сообщениями
- Автоматическое восстановление после сбоев

### 5.5 Совместимость
- iOS: версия 13.0 и выше
- Android: версия 8.0 (API 26) и выше
- Поддержка основных размеров экранов (телефоны, не планшеты в MVP)

### 5.6 Масштабируемость
- Архитектура должна поддерживать подключение к различным бэкенд-системам
- Возможность добавления новых функций без существенной переработки

## 6. Технический стек и архитектура

### 6.1 Технологии
- **Frontend Framework:** React Native
- **Backend:** Node.js
- **Стилизация:** Tailwind CSS
- **UI Kit:** shadcn/ui

### 6.2 Архитектура приложения
Трехуровневая архитектура:

1. **View Layer (Слой представления)**
   - Экраны приложения
   - UI-компоненты
   - Навигация

2. **Domain Layer (Слой домена)**
   - Бизнес-сущности (Product, Portfolio, Message, User)
   - Бизнес-логика
   - Валидация

3. **Adapters Layer (Слой адаптеров)**
   - API-клиенты для взаимодействия с бэкендом
   - Маппинг данных
   - Кэширование

### 6.3 Интеграции
- REST API бэкенда управляющей компании
- Push-уведомления (Firebase Cloud Messaging / Apple Push Notifications)
- Сервис аутентификации

## 7. Ограничения и предположения

### 7.1 Ограничения
- Только мобильные платформы (iOS, Android)
- React Native как обязательная технология
- Нет веб-версии в MVP
- Нет планшетной версии в MVP

### 7.2 Предположения
- Бэкенд API готов или будет готов к моменту интеграции
- Дизайн-макеты будут предоставлены до начала разработки
- Тестовые учетные данные для QA будут доступны
- Доступ к существующей системе управления активами для интеграции

### 7.3 Зависимости
- API документация от бэкенд-команды
- Дизайн-система и макеты
- Доступ к тестовой среде бэкенда
- Учетные записи для тестирования чата

## 8. Критерии приемки

### 8.1 Определение готовности (Definition of Done)
- [ ] Код написан и соответствует стандартам разработки
- [ ] Unit-тесты написаны и проходят успешно
- [ ] Integration-тесты для критических сценариев
- [ ] Код прошел code review
- [ ] UI соответствует дизайн-макетам
- [ ] Функциональность протестирована QA-командой
- [ ] Документация обновлена
- [ ] Приложение готово к развертыванию

### 8.2 Критерии успеха MVP
- Все Must Have функции реализованы и работают
- Приложение стабильно работает на целевых устройствах
- Критические баги отсутствуют
- Пользователи могут выполнять основные сценарии использования
- Код покрыт тестами (минимум 70% coverage)

---

## 9. Вне рамок (Out of Scope)

### 9.1 Не входит в MVP
- Торговые операции через приложение
- Веб-версия приложения
- Версия для планшетов
- Интеграция с банковскими системами для переводов
- Самостоятельное изменение состава портфеля клиентом
- Алгоритмические рекомендации по инвестициям
- Социальные функции (рейтинги, отзывы)
- Мультиязычность (только русский язык в MVP)

## 10. Риски и митигация

| Риск | Вероятность | Влияние | Митигация |
|------|-------------|---------|-----------|
| Задержка готовности бэкенд API | Средняя | Высокое | Раннее согласование контрактов API, использование моков на начальном этапе |
| Изменения требований в процессе разработки | Средняя | Среднее | Гибкая методология, приоритизация изменений |
| Проблемы с интеграцией существующих систем | Средняя | Высокое | Ранний этап интеграции, технический спайк |
| Проблемы с безопасностью | Низкая | Критическое | Security audit, penetration testing |
| Низкое качество связи при чате | Низкая | Среднее | Офлайн-очередь сообщений, retry-механизмы |

## 11. Открытые вопросы

### Требующие уточнения:
1. **Аутентификация:**
   - Регистрация через приложение или только после договора?
   - Двухфакторная аутентификация: обязательна или опционально?
   - Таймаут автоматического выхода из приложения?

2. **Бэкенд и интеграции:**
   - Готовность API бэкенда?
   - Формат данных (JSON Schema)?
   - Механизм real-time обновлений для чата (WebSocket, polling)?

3. **Дизайн:**
   - Есть ли существующий брендбук/дизайн-система?
   - Кто предоставляет дизайн-макеты?

4. **Правовые и регуляторные требования:**
   - Специфические требования регуляторов?
   - Необходимость отображения дисклеймеров?
   - Требования к хранению персональных данных?

5. **Чат:**
   - Поддержка медиа-файлов в чате?
   - История сообщений: глубина хранения?
   - Режим работы менеджеров (ответ в рабочее время)?
